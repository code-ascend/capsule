# capsule

Инструмент для создания портативных Linux-контейнеров из OCI-образов. Результат — единый ELF-бинарник со статически
собранным Go-рантаймом внутри.

## Особенности

- **Один файл** — скачал и запустил, нет зависимостей
- **Портативность** — можно положить куда угодно, например на USB-флеш-накопитель
- **Без root** — для запуска капсулы не нужны права суперпользователя
- **Бесшовный опыт** — приложения читают и хранят конфиги в вашем `$HOME`
- **Чистая система** — приложение работает в своём окружении со своими библиотеками, не засоряет хост
- **Набор приложений** — одна капсула — целый набор приложений и утилит
- **Статический Go-рантайм** — без зависимости от системной glibc/bash
- **Изолированный рантайм** — утилиты работают через встроенный ld-linux и libc

## Установка

### Установка из ALS

> Сами капсулы обладают большой автономностью и портативностью, но для их сборки требуется данный проект

```bash
apt-repo add rpm https://altlinux.space/api/packages/dmitry/alt/group/capsule-nightly/sisyphus.repo x86_64 classic
apt-get update
apt-get install capsule
ИЛИ
sudo apm repo add rpm https://altlinux.space/api/packages/dmitry/alt/group/capsule-nightly/sisyphus.repo _arch_ classic
sudo apm s update
sudo apm s install capsule
```

### Установка вручную

```bash
sudo apt-get install squashfs-tools go meson ninja-build gettext \
                     libgpgme-devel libbtrfs-devel libdevmapper-devel \
                     shadow-submap fuse-overlayfs containers-common netavark
meson setup build --prefix=/usr
meson compile -C build
meson install -C build
```

### Обновление переводов

```bash
meson compile -C build pot-update
```

## Пример сборки портативной капсулы

> Внимание! Это пример сборки max, в `examples` есть другие примеры с разным набором рантаймов и пакетов.
> Цель данного примера - ознакомление со сборкой капсул. Конечная задача после ознакомления - создать собственную
> капсулу с нужным набором пакетов.

```bash
# Сборка max (путь к файлу или к ресурсу с инструкцией сборки)
capsule build https://altlinux.space/dmitry/capsule/raw/branch/main/examples/max.yaml -v
# Справка
max -h
# Запуск
max
# Войти внутрь, все изменения будут сохранены
max shell
# Экспорт ярлыков
max export
# Убрать ярлыки из хост системы
max unexport
```

## Особенности работы с капсулами

Все изменения, сделанные внутри капсулы (установка пакетов, настройка конфигов), сохраняются между запусками и могут
быть зафиксированы в образе.

```bash
# Войти в shell, установить новый пакеты, изменить файлы
max shell
# Войти с повышенными правами (для работы пакетных менеджеров)
sudo max shell
# Зафиксировать изменения
max commit
# Сбросить все изменения (те, что не были зафиксированы через commit)
max clean
```

## Запуск хост-команд из капсулы (host-exec)

Капсула изолирована от хоста, но иногда из-под неё нужно выполнить команду **на самом хосте** — например, чтобы браузер
из капсулы открыл ссылку в системном просмотрщике, а не пытался сделать это своими силами внутри. Для этого в капсулу
встроен мост `capsule-host-exec`.

Включается в `config.yaml`:

```yaml
host_exec: true
```

После сборки рантайм при запуске поднимает abstract-UNIX-сокет на хосте и пробрасывает внутрь капсулы клиент
`capsule-host-exec`, который шлёт через сокет имя команды и аргументы на хост-сторону, где они исполняются как обычный
процесс под вашим UID.

Дополнительно тот же клиент автоматически биндится **под несколькими алиасами**, чтобы перехватить типовые
внутрикапсульные обращения наружу:

| алиас в капсуле     | поведение                                                              |
|---------------------|------------------------------------------------------------------------|
| `capsule-host-exec` | прямой вызов: `capsule-host-exec CMD [ARGS...]`                        |
| `xdg-open`          | ссылки/файлы открываются в хостовом окружении (браузер, файл-менеджер) |
| `gio`               | gio-операции (mount/open/...) уходят на хост                           |

## Оверрайды конфига изнутри rootfs

Install-скрипты (или установленные ими пакеты) могут положить на корень rootfs файл `.capsule.overrides.yml` и тем
самым дополнить манифест капсулы изнутри. После завершения install-шагов и до запечатывания squashfs capsule
смерживает этот файл поверх билд-конфига и удаляет его, чтобы он не попал в итоговый образ. Удобно,
когда установленный пакет хочет сам объявить свой `launch`, `export`, `env` или другие верхнеуровневые поля без
редактирования внешнего YAML.

```yaml
# /.capsule.overrides.yml внутри rootfs
launch: /usr/bin/myapp
export:
  apps:
    - desktop: /usr/share/applications/myapp.desktop
```

## Как это работает

Capsule — это единый исполняемый файл со следующей структурой:

| Go runtime + utils.tar.gz | binconfig (JSON) | SquashFS | Footer  |
|---------------------------|------------------|----------|---------|
| ~14 MB                    | сотни байт       | образ    | 32 байт |

## По Пунктам

1. Go-рантайм при запуске читает `/proc/self/exe`, парсит 32-байтовый footer и узнаёт смещения встроенных полезных
   нагрузок (binconfig + squashfs).
2. Из себя же распаковывает `utils.tar.gz` (bwrap, squashfuse, unionfs, mksquashfs, ld-linux, либы) во временный
   workspace.
3. Squashfuse монтирует SquashFS-образ прямо из бинарника по смещению (-o offset=N) через FUSE — без root, в userspace.
4. Опционально поверх кладётся записываемый unionfs-overlay, в который сливаются изменения сессии и перенесённые с хоста
   NVIDIA-библиотеки.
5. Bubblewrap создаёт изолированное окружение через user namespaces: корень = squashfs (или overlay), пробрасывает
   `$HOME`, `/dev`, `/tmp` (включая X11-сокеты `/tmp/.X11-unix`), `/run` (включая Wayland-сокет
   `/run/user/UID/wayland-*` и пользовательские unix-сокеты), накладывает поверх `/etc/resolv.conf`, `/etc/hosts`,
   `/etc/localtime`, `/etc/machine-id` с хоста.
6. Если в манифесте `host_exec: true` — рантайм поднимает abstract-UNIX-сокет и биндит клиент `capsule-host-exec`
   (плюс алиасы `xdg-open`, `gio`) внутрь капсулы для проброса команд обратно на хост.
